## 1. 프로젝트 개요

| 항목 | 내용 |
|---|---|
| **프로젝트명** | IBM Credit Card Transaction 유저 행동 분석 |
| **진행 기간** | 2026.03 ~ 2026.04 (약 2개월) |
| **팀 구성** | **개인(solo) 프로젝트** — 전 과정 단독 수행 (데이터 적재 → 전처리 → 분석 → 시각화 → 문서화) |

### 1.1 프로젝트 배경
- 핀테크 DA 직무를 목표로, **"전처리부터 대시보드까지 end-to-end 파이프라인"**을 단독으로 구현할 수 있음을 증명하기 위한 포트폴리오 프로젝트.
- 실데이터 공개가 어려운 금융 도메인 특성상, **공개 가능한 합성 데이터(Apache 2.0)**를 선택해 포트폴리오 공개 리스크를 제거.
- 단순 시각화가 아니라 **그로스 분석 프레임워크(AARRR·Cohort·LTV·RFM)**와 **통계 검정(A/B Test)**을 결합해, "지표 설계 → 가설 → 검증 → 비즈니스 제언"의 완결된 분석 사이클을 보여주는 데 초점.

### 1.2 프로젝트 목표
1. 핀테크 서비스 유저의 **생애주기(lifecycle)**를 데이터로 파악.
2. **서비스 개선 가설**을 도출하고, **통계적으로 검증**.
3. **데이터 기반 의사결정 프레임워크**를 구축해 비즈니스 제언까지 연결.

### 1.3 해결하고자 한 비즈니스 문제
> **"유저가 어느 단계에서 이탈하며, 어디에 개입하면 LTV를 가장 효율적으로 끌어올릴 수 있는가?"**

이 질문을 AARRR 퍼널로 병목을 찾고 → Cohort/LTV/RFM으로 유저 가치를 정량화 → A/B Test로 개입 효과를 검증하는 흐름으로 풀어냄.

💡 **핵심**: "분석을 위한 분석"이 아니라 **"하나의 비즈니스 질문"을 향해 5개 분석 기법이 유기적으로 연결**되어 있다는 점이 차별점.

## 2. 프로젝트 전체 타임라인

분석은 다음 순서로 진행됨. (개인 프로젝트이므로 각 단계는 단독 의사결정으로 진행)

```
[1] 문제 정의      핀테크 유저 lifecycle / LTV 개선 레버 탐색
       │
[2] 데이터 수집·DB 구축   2.18GB CSV → chunk 스트리밍 → SQLite 적재
       │
[3] 데이터 탐색(EDA)      결측·이상치·사기비율·MCC 분포·연도 추이
       │
[4] 전처리 의사결정       분석용 View 2개 설계 (clean / ltv)
       │
[5] 가설 설정 & 분석      AARRR → Cohort → LTV → RFM 순차 수행
       │
[6] 가설 검증            A/B Test (Power Analysis → 시뮬레이션 → z-test)
       │
[7] 인사이트 도출        비즈니스 임팩트 정량 추정 ($6.6M)
       │
[8] 시각화 & 문서화       Tableau 6시트 통합 대시보드 + 기술보고서
```

💡 **EDA → 전처리 의사결정 → 분석** 순서가 명확하고, "왜 이 전처리를 했는가"가 분석 설계와 연결됨 (3·4번 참조).

### 3.2 데이터의 특징

## 4. 분석 과정 상세 정리

각 단계: **무엇을 / 왜 / 대안 / 선택 이유 / 결과** 구조.

### 4.1 데이터 적재 (DB 구축)
- **무엇을**: 2.18GB(2,400만 행) CSV를 SQLite에 적재.
- **왜**: 한 번에 메모리 로드 시 RAM 16GB 부족 발생.
- **대안**: (a) 전체 로드 → 실패, (b) Dask/Spark 도입, (c) chunk 스트리밍.
- **선택 이유**: 개인 환경에서 추가 인프라 없이 가능한 **chunk 스트리밍**(`chunksize=100,000`) 채택 → SQLite append.
- **결과**: 24,386,900행 적재 완료, 메모리 안정.

💡 **대용량 데이터를 "로컬 자원 제약 안에서" 처리한 엔지니어링 판단** — 실무 면접 단골 주제.

### 4.2 EDA & 전처리 의사결정
- **무엇을**: 결측·이상치·사기비율·MCC 분포·연도 추이 탐색 후 분석용 View 설계.
- **왜**: 어떤 거래를 "분석 대상"으로 볼지 정의해야 이후 모든 분석의 기준이 정해짐.
- **핵심 의사결정 (근거 명시형)**:

| 항목 | 처리 | 근거 |
|---|---|---|
| 사기 거래 | 전체 제외 | 유저가 의도한 행동이 아님 |
| 2020년 | 제외 | 1~2월만 존재해 연간 비교 불가 |
| 음수 Amount(환불) | LTV 제외 / 행동분석 포함 | LTV는 순수 지출 기준, 행동분석은 "거래 발생" 자체가 신호 |
| 오류 거래 | LTV 제외 | 미완료 거래는 실제 수익으로 보기 어려움 |
| Amount > $5,000 | LTV 제외 | 이상치 17건이 평균 왜곡 |

- **결과**: View 2개로 분리 → `clean_transactions`(사기·2020 제외, 행동분석용), `ltv_transactions`(추가로 환불·오류·이상치 제외, 금액분석용).

💡 **"같은 데이터라도 분석 목적에 따라 다른 기준 적용"** — 환불 거래를 LTV에선 빼고 행동분석엔 넣은 판단이 분석 성숙도를 보여줌.

### 4.3 AARRR 퍼널 분석
- **무엇을**: 유저 lifecycle 단계별 전환율 측정.
- **왜**: 어느 단계가 병목인지 찾아 개입 지점을 특정하기 위해.
- **대안**: 표준 AARRR 그대로 적용 vs 데이터 특성에 맞게 재정의.
- **선택 이유**: 가입 데이터·추천 데이터가 없는 카드 거래 데이터 특성상 **프레임워크를 재정의**.

| 단계 | 본 분석 정의 | 결과 |
|---|---|---|
| Acquisition | 첫 거래 발생 유저 | 1,657명 |
| Activation | 첫 거래 후 30일 내 재거래 | 1,093명 (66.0%) |
| Retention | Activation 유저 중 다음 연도 재거래 | 1,066명 (97.5%) |
| Revenue | 카드 특성상 Acquisition과 동일 | 1,657명 |
| Referral | 데이터 없음 → 스킵 | - |

- **결과**: **Activation(66%)이 유일한 이탈 구간**. Activation된 유저의 **97.5%가 장기 전환** → 초기 30일이 lifecycle의 핵심 분기점.
- **유저 4분류**: 진성 유저 64.3% / 느린 정착형 31.3% / 이탈 2.8% / 초반 반짝 1.6%.

### 4.4 Cohort 분석
- **무엇을**: 첫 거래 연도 기준 코호트별 거래 빈도·금액 변화 추적.
- **왜**: 시간 경과에 따른 유저 행동 변화를 보기 위해.
- **대안**: 표준 Retention Matrix(재방문율).
- **선택 이유**: 카드 데이터 특성상 단순 재방문율이 **95%+로 변별력이 없어**, **"연간 거래 빈도·금액 변화"로 지표를 재정의**.
- **결과**: 1년차 950회/년 → 17년차 1,227회/년 (**+29%**). 카드 사용이 습관화될수록 사용량 증가.
- **이상치 해석**: 2011 코호트(48명)·2012 코호트(26명)는 **소규모 샘플 노이즈**로 판단(과대해석 방지).
- **0년차 제외 근거**: 첫 거래 연도는 부분 기간(예: 11월 첫 거래 시 2개월치)이라 완전한 1년과 비교 시 편향 → 1년차부터 분석.

💡 **"지표가 변별력이 없으면 지표 자체를 바꾼다"** — 무비판적 프레임워크 적용을 거부한 분석적 판단.

### 4.5 LTV 분석
- **무엇을**: 유저 생애가치 산출.
- **왜**: 유저별 사용 기간이 달라(2002년 가입자 17년 vs 2018년 가입자 1년) 단순 누적 금액 비교는 불공평.
- **대안**: (a) 단순 누적 금액, (b) 측정형 LTV(기간 보정), (c) 예측형 LTV.
- **선택 이유**: 관찰 기간 편향을 보정한 **측정형 LTV = 총 거래금액 ÷ 활동 연수** 채택.
- **결과**: 중앙값 **$43,170**, 평균 $50,507, 상위 25% $63,836+, **우편향**(소수 헤비유저가 평균 견인).
- **예측형 LTV는 의도적으로 보류**: 코호트별 관찰 기간 차이로 편향 추정값 발생 확인 → 실무에선 BG/NBD·Kaplan-Meier 필요함을 명시.

💡 **"안 되는 분석을 억지로 하지 않고, 왜 안 되는지·무엇으로 보완해야 하는지 명시"** — 정직성과 방법론 이해도를 동시에 보여줌.

### 4.6 RFM 세그멘테이션
- **무엇을**: Recency·Frequency·Monetary 점수화 후 5개 세그먼트 분류.
- **왜**: 유저를 가치별로 나눠 차별적 전략을 세우기 위해.
- **데이터 특성 대응**: 대부분 유저가 2019년까지 활동 → `recency_years=0`이 다수라 `qcut` 5분위 분할 불가 → **R은 구간별 직접 점수 부여**(0년=5, 1~2년=4, 3~5년=3, 6~9년=2, 10년+=1). F·M은 5분위.
- **결과**:

| 세그먼트 | 기준(RFM) | 유저 수 | 비율 | 평균 연간 거래금액 |
|---|---|---|---|---|
| VIP | 13점+ | 421명 | 31.5% | $86,667 |
| Loyal | 10~12점 | 449명 | 33.6% | $46,426 |
| Potential | 7~9점 | 434명 | 32.5% | $25,514 |
| At Risk | 6점↓ + R≤2 + (F≥3 or M≥3) | 2명 | 0.1% | $43,621 |
| Dormant | 6점↓ + R≤2 + F<3 & M<3 | 31명 | 2.3% | $15,816 |

- **행동 패턴**: VIP vs Potential 최대 차이는 **교통/자동차(27.8% vs 17.0%, +10.8%p)**. 식품/식료품은 전 세그먼트 31~34%로 차이 없음(필수 지출). 월별 패턴은 전 세그먼트 공통 **2월 최저·12월 최고**.

### 4.7 A/B 테스트
- **무엇을**: "초기 30일 리마인드 알림"의 Activation 개선 효과 검증.
- **왜**: AARRR에서 찾은 병목(Activation)에 대한 개입 가설을 통계로 입증하기 위해.
- **설계**: 단측 z-test (방향성 명확), α=0.05, power=0.80.
- **Power Analysis**: p_control 0.66, p_treatment 0.75(MDE +9%p) → Cohen's h = 0.1979(small) → 그룹당 316명, 전체 632명 필요. 보유 1,657명으로 **표본 충분**.
- **검정 방식 선택 이유**: 측정 지표가 **비율(전환율)**이므로 비율 검정(z-test) 적합. (t-test는 거래금액 등 연속형 평균 비교용 — 구분 명시.)
- **시뮬레이션 2중 설계 + 교차 검증**:

| 방식 | 전환 확률 근거 | 실험군 전환율 | Z | p-value | 결론 |
|---|---|---|---|---|---|
| 임의 설정 | 분석자 판단(Potential 40/Loyal 20/VIP 10%) | 79.1% | 3.2415 | 0.0006 | 기각 |
| 벤치마크 기반 | 업계 사례값(Potential 25/Loyal 18/VIP 8%) | 75.6% | 2.2071 | 0.0137 | 기각 |

- **결과**: 두 시뮬레이션 모두 α=0.05에서 귀무가설 기각. **보수적 벤치마크 기반에서도 유의** → 교차 검증 완료.
- **벤치마크 출처**: whmarketing.org(재활성화 25%), Appier(개인화 푸시 반응 4배), ditoday.com(이탈위험 고객 구매 +134%).

💡 **실제 알림 로그가 없는 한계를 "임의 + 벤치마크 이중 시뮬레이션"으로 우회하고 교차 검증** — 제약 조건 하에서 결론의 강건성을 확보한 설계.

|---|---|---|
| **AARRR 재정의** | 가입/추천 기반 표준 퍼널 | 카드 데이터에 맞게 Acquisition=첫거래, Revenue=Acquisition, Referral 스킵으로 재정의 |

## 7. 기술적 성장

**데이터 시각화**
- Tableau Public 6개 시트 → 1개 통합 대시보드 구성.

**분석 방법론**

## 12. 면접 활용 포인트 (STAR)

**STAR ① — 대용량 처리 (기술 문제 해결)**
- S: 2,400만 행을 로컬(16GB)에서 분석해야 함.
- T: 메모리 초과 없이 전량 적재.
- A: chunk 스트리밍(10만 단위)으로 SQLite에 순차 적재.
- R: 24,386,900행 안정 적재, 이후 모든 분석의 기반 확보.

**STAR ② — 지표 재정의 (문제 정의)**
- S: 표준 Cohort Retention이 95%+로 변별력 없음.
- T: 의미 있는 유저 변화 지표 필요.
- A: "연간 거래 빈도·금액 변화"로 지표 재설계.
- R: 1년차 950회 → 17년차 1,227회(+29%)라는 습관화 패턴 발견.

**STAR ③ — 분석 보류 판단 (정직성·방법론)**
- S: 예측형 LTV를 시도.
- T: 신뢰 가능한 LTV 산출.
- A: 관찰 기간 편향을 발견하고 측정형 LTV로 전환, 한계와 보완책(BG/NBD) 명시.
- R: 과대추정 회피, 보고서 신뢰도 확보.

**STAR ④ — 제약 하 검증 설계 (창의성)**
- S: 실제 알림 로그가 없어 A/B 불가능해 보임.
- T: 그래도 개입 효과를 검증.
- A: 임의 + 벤치마크 이중 시뮬레이션으로 교차 검증.
- R: 두 케이스 모두 유의(p=0.0006/0.0137), 강건한 결론.
